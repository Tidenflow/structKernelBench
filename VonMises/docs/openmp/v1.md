# Von Mises / OpenMP / v1

## 当前实现

OpenMP v1 保留 baseline 的逐元素公式，只在外层元素循环上加并行：

```cpp
#pragma omp parallel for
for (int i = 0; i < N; ++i) {
    float ds = sx[i] - sy[i];
    float ss = sy[i] - sz[i];
    float ts = sz[i] - sx[i];
    vm[i] = std::sqrt(0.5f * (ds*ds + ss*ss + ts*ts +
                               6.0f * (txy[i]*txy[i] + tyz[i]*tyz[i] + tzx[i]*tzx[i])));
}
```

源码位置：`VonMises/kernels/openmp.cpp`

## 优化程度

这是 v1 的 CPU 多线程实现。它没有改变数据布局，也没有拆分公式，只把独立元素分配给多个 CPU 线程。

| 方向 | 当前状态 |
|---|---|
| 线程级并行 | `#pragma omp parallel for` 按元素并行 |
| 元素内计算 | 与 baseline 相同 |
| 调度策略 | OpenMP 默认调度 |
| 数据重排 | 无 |
| 适合目标 | 观察多核 CPU 对纯逐元素计算的提升 |

## 优化思路

Von Mises 的每个输出 `vm[i]` 只依赖同一个 `i` 上的六个输入分量，所以元素之间没有写冲突，也不需要归约或锁。

OpenMP v1 利用这个特点，把 `i = 0..N-1` 的循环切给多个线程。相比 SpMV，这种逐元素算子更适合 OpenMP，因为：

- 每个元素工作量接近一致。
- 输入输出都是连续数组。
- 没有稀疏矩阵带来的不规则访存。
- 每个线程写不同的 `vm[i]`。

## 性能瓶颈

OpenMP 版本的瓶颈主要取决于 CPU 核数、内存带宽和 `sqrt` 吞吐：

- 小规模数据可能被线程启动和调度开销抵消。
- 大规模数据会更接近内存带宽和数学函数吞吐上限。
- 如果编译器同时对线程内部做向量化，OpenMP 可能获得更好的复合加速。

## 观察重点

比较 baseline 和 OpenMP 时，可以重点看吞吐量是否随元素数增大而稳定提升。对于这种连续逐元素后处理算子，OpenMP 通常比在不规则 CSR-SpMV 上更容易获得可见加速。
