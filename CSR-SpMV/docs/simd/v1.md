# CSR-SpMV / SIMD / v1

## 当前实现

SIMD v1 保持 CSR 计算结构不变，核心仍然是每行一个内层归约：

```cpp
for (int i = 0; i < N; ++i) {
    double sum = 0.0;
    int start = row_ptr[i];
    int end = row_ptr[i + 1];
    for (int k = start; k < end; ++k) {
        sum += values[k] * x[col_idx[k]];
    }
    y[i] = sum;
}
```

源码位置：`CSR-SpMV/kernels/simd.cpp`

## 优化程度

这是自动向量化方向的 v1。当前实现没有写 AVX/SSE intrinsic，也没有改成 ELL、SELL-C-sigma 这类更适合 SIMD 的稀疏格式；它主要依赖编译参数：

- `-O3`
- `-march=native`
- `-ffast-math`

| 方向 | 当前状态 |
|---|---|
| SIMD 方式 | 编译器自动向量化尝试 |
| 显式 intrinsic | 无 |
| 数据格式转换 | 无，仍然使用 CSR |
| 线程并行 | 无，单线程 |
| 适合目标 | 观察 CSR 内层循环能被编译器优化到什么程度 |

## 优化思路

这个版本的核心思想是把内层循环写得尽量直接，让编译器更容易识别“乘加归约”模式：

```cpp
sum += values[k] * x[col_idx[k]];
```

其中 `values[k]` 和 `col_idx[k]` 都是连续访问，编译器可以较容易地生成顺序 load。但 `x[col_idx[k]]` 是 gather 型间接访存，是否能形成有效 SIMD 取决于 CPU 指令集、编译器判断和矩阵列索引分布。

## 关键限制

CSR 对 SIMD 并不天然友好：

- 每行长度可能不同，内层循环边界不固定。
- `sum` 是归约变量，存在循环携带依赖。
- `x[col_idx[k]]` 不是连续读取，向量 gather 成本较高。
- 如果每行非零元很少，向量 lanes 很难被充分填满。

因此，SIMD v1 的性能提升可能明显小于 OpenMP 或 cuSPARSE。它更像是在“不改变数据结构”的前提下，测试编译器能帮到多少。

## 应该怎么看结果

如果 SIMD 比 baseline 快，通常说明编译器在内层归约、展开、调度或局部寄存器使用上做了有效优化。

如果 SIMD 提升很小，也不代表 SIMD 思路无效，而是说明标准 CSR 的间接访存和不规则行长限制了自动向量化。更激进的 SIMD 版本通常会考虑：

- 固定或分桶每行非零元数量。
- 使用 ELL/SELL 类格式改善对齐和连续性。
- 手写 AVX gather 或分块归约。

这些都还不属于当前 v1 的实现范围。
