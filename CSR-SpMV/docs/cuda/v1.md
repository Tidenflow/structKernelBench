# CSR-SpMV / CUDA / v1

## 当前实现

CUDA v1 使用“一行一个 GPU 线程”的映射方式：

```cpp
int i = blockDim.x * blockIdx.x + threadIdx.x;
if (i >= N) return;

double sum = 0.0;
int start = row_ptr[i];
int end = row_ptr[i + 1];
for (int k = start; k < end; ++k) {
    sum += values[k] * x[col_idx[k]];
}
y[i] = sum;
```

源码位置：`CSR-SpMV/kernels/cuda_kernel.cu`

## 优化程度

这是 v1 的直观 GPU 并行实现。它把 CPU baseline 的行级独立性搬到 GPU 上，但没有做 warp 级协作、共享内存缓存、行分桶或向量化读取。

| 方向 | 当前状态 |
|---|---|
| 线程映射 | 一个 CUDA thread 负责一行 |
| block 大小 | `256` threads |
| 内存管理 | 每次调用分配、拷贝、释放 |
| kernel 计时 | 使用 CUDA event 统计 kernel 时间 |
| transfer 计时 | 单独统计 H2D + D2H，但 UI 当前主要展示 kernel 时间 |

## 优化思路

CSR-SpMV 的每行输出相互独立，所以最直接的 GPU 映射就是每个线程计算一个 `y[i]`。这样实现简单，逻辑和 CPU baseline 对齐，适合做手写 CUDA 的第一条性能曲线。

当前 wrapper 做了这些事情：

- 在 device 上分配 `values`、`col_idx`、`row_ptr`、`x`、`y`。
- 将 CSR 数据和输入向量从 host 拷贝到 device。
- 启动 `spmv_cuda_kernel`。
- 将 `y` 从 device 拷回 host。
- 用 CUDA event 分别统计传输时间和 kernel 时间。

## 性能瓶颈

这个版本很容易理解，但不是高性能 SpMV 的最终形态：

- 如果某些行非零元很多，一个线程会独自处理长行，负载不均明显。
- 如果每行非零元很少，单线程工作量太小，GPU 吞吐可能不充分。
- `x[col_idx[k]]` 仍然是不规则读取，global memory 合并访问效果有限。
- 每次 benchmark 调用都重新分配和拷贝数据，端到端时间会被传输和分配影响。
- 一个线程内部串行归约，未使用 warp 内并行归约。

## 观察重点

这个版本适合回答一个很具体的问题：仅仅把“行级并行”搬到 GPU 上，能获得多少 kernel 吞吐提升？

如果它和 cuSPARSE 差距很大，通常说明工业库在行分组、负载均衡、访存合并、向量化策略和内部调度上做了更多工作。
