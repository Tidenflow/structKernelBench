# CSR-SpMV / OpenMP / v1

## 当前实现

OpenMP 版本保留 baseline 的 CSR 内层循环，只在最外层行循环上增加并行：

```cpp
#pragma omp parallel for
for (int i = 0; i < N; ++i) {
    double sum = 0.0;
    int start = row_ptr[i];
    int end = row_ptr[i + 1];
    for (int k = start; k < end; ++k) {
        sum += values[k] * x[col_idx[k]];
    }
    y[i] = sum;
}
```

源码位置：`CSR-SpMV/kernels/openmp.cpp`

## 优化程度

这是 v1 的直接多线程实现。它没有改变算法和数据布局，只把不同行分配给多个 CPU 线程。

| 方向 | 当前状态 |
|---|---|
| 线程级并行 | `#pragma omp parallel for` 按行并行 |
| 行内优化 | 与 baseline 相同 |
| 调度策略 | 使用 OpenMP 默认调度 |
| 数据重排 | 无 |
| 适合目标 | 快速观察多核 CPU 对 SpMV 的提升 |

## 优化思路

CSR-SpMV 中每一行 `y[i]` 的计算相互独立，因此行循环天然可以并行。OpenMP v1 利用这一点，把不同的 `i` 分配给不同线程。

这个实现的优点是侵入性低：

- 不需要改 CSR 数据结构。
- 不需要改 benchmark runner。
- 和 baseline 的代码形态几乎一致，便于对照。
- 输出 `y[i]` 每行独立写入，不需要原子操作或锁。

## 性能瓶颈

OpenMP 版本不一定随线程数线性加速，主要原因是 SpMV 很快会碰到内存系统瓶颈：

- 多线程同时读取 `values`、`col_idx`、`x`，内存带宽压力上升。
- `x[col_idx[k]]` 是间接访存，缓存命中率取决于稀疏矩阵结构。
- 每行非零元数量如果差异很大，默认静态分配可能出现负载不均。
- 行很短时，线程调度开销和内层循环工作量相比会变得明显。

## 观察重点

在界面中比较 baseline 和 OpenMP 时，可以重点看：

- 小规模矩阵是否因为线程开销导致收益有限。
- 中大规模矩阵是否出现明显加速。
- 继续增大规模后吞吐量是否趋于平台内存带宽上限。
