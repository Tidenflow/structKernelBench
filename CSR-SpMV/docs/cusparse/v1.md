# CSR-SpMV / cuSPARSE / v1

## 当前实现

cuSPARSE v1 使用 NVIDIA 官方库的 `cusparseSpMV` API 作为工业级参考实现。当前代码仍然接收项目自己的 CSR 三数组，然后在 wrapper 中创建 cuSPARSE 描述符：

```cpp
cusparseCreateCsr(
    &matA, N, N, nnz,
    d_row_ptr, d_col_idx, d_values,
    CUSPARSE_INDEX_32I, CUSPARSE_INDEX_32I,
    CUSPARSE_INDEX_BASE_ZERO, CUDA_R_64F);

cusparseSpMV(
    cusparse_handle, CUSPARSE_OPERATION_NON_TRANSPOSE,
    &alpha, matA, vecX, &beta, vecY,
    CUDA_R_64F, CUSPARSE_SPMV_ALG_DEFAULT, d_buffer);
```

源码位置：`CSR-SpMV/kernels/cusparse_kernel.cu`

## 优化程度

这是当前项目中 SpMV 的 GPU 工业库参考，不是手写 kernel。它代表“在相同输入 CSR 数据下，成熟库能做到什么水平”。

| 方向 | 当前状态 |
|---|---|
| 核心计算 | `cusparseSpMV` |
| 数据格式 | CSR |
| 算法选择 | `CUSPARSE_SPMV_ALG_DEFAULT` |
| handle | 静态持久化，避免每次重建 |
| buffer | 每次查询并分配临时 buffer |

## 优化思路

cuSPARSE 内部会根据矩阵描述、数据类型、索引类型和算法选项选择合适实现。它可能使用比“一行一个线程”更复杂的策略，例如：

- 对不同长度的行使用不同处理路径。
- 使用 warp 或 block 协作处理长行。
- 改善访存合并和缓存利用。
- 使用临时 buffer 保存中间调度或归约数据。
- 针对 NVIDIA GPU 架构做专门优化。

这些细节由库内部管理，项目代码只负责准备合法的 CSR 描述符和输入输出向量。

## 与手写 CUDA 的区别

手写 CUDA v1 的目标是清楚展示 GPU 行级并行的基本形态；cuSPARSE 的目标是高性能和鲁棒性。

因此比较二者时，重点不是“哪段代码更短”，而是：

- 手写 kernel 距离工业库还有多少差距。
- 当前矩阵规模和 `nnz/row` 下，库是否明显更能吃满 GPU。
- 随着矩阵变大，cuSPARSE 的吞吐曲线是否更稳定。

## 当前 wrapper 的额外成本

当前项目为了 benchmark 接口统一，每次调用仍会执行 host-device 拷贝、descriptor 创建、buffer 分配和结果回拷。UI 展示的主要计时使用 kernel/library 调用时间，不把全部端到端成本都混在一起。

如果未来研究端到端场景，可以单独展示 transfer time 和 total time。
