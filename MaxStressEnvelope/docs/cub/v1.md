# MaxStressEnvelope / CUB / v1

## 当前实现

CUB v1 把多工况包络拆成两个阶段：

1. 先用 `vm_all_kernel` 计算所有 `M * N` 个工况的 Von Mises 应力，写入 `d_vm_all`。
2. 再用 `cub::DeviceSegmentedReduce::Max` 对每个单元的连续工况段做最大值归约。

关键调用：

```cpp
cub::DeviceSegmentedReduce::Max(d_temp, temp_bytes,
                                 d_vm_all, d_envelope,
                                 M, d_offsets, d_offsets + 1);
```

其中 `d_offsets[i] = i * N`，所以第 `i` 个 segment 正好对应第 `i` 个单元的 `N` 个工况。

源码位置：`MaxStressEnvelope/kernels/cub_kernel.cu`

## 优化程度

这是当前项目中 MaxStressEnvelope 的 GPU 工业级原语参考实现。它不使用手写融合归约，而是把最大值归约交给 CUB 的分段归约原语。

| 方向 | 当前状态 |
|---|---|
| Phase 1 | 手写 kernel 计算所有工况 Von Mises |
| Phase 2 | CUB `DeviceSegmentedReduce::Max` |
| segment 定义 | 每个单元一个 segment，长度为 `N` |
| 中间数组 | `d_vm_all`，大小为 `M * N` |
| 临时空间 | CUB 查询并分配 `d_temp` |

## 优化思路

多工况包络可以看作标准 segmented reduction：

```text
for each element i:
    envelope[i] = max(vm_all[i*N : (i+1)*N])
```

CUB v1 利用这个结构，把“取 max”交给成熟的 GPU 原语。这样可以观察工业级分段归约对这个问题的表现，也能和手写 CUDA 融合 kernel 形成对照。

## 与手写 CUDA 的区别

手写 CUDA v1 是融合式：

- 每个线程处理一个单元。
- Von Mises 和 max 在同一个 kernel 中完成。
- 不写出 `M*N` 个中间值。

CUB v1 是分阶段：

- Phase 1 先计算所有工况的 Von Mises。
- Phase 2 使用 CUB 对每个 segment 做 max。
- 需要额外的 `d_vm_all` 中间数组和 CUB 临时 buffer。

因此 CUB 不一定在所有规模上都更快。它展示的是“使用工业级归约原语”的路径，而 CUDA v1 展示的是“针对当前公式做融合”的路径。

## 性能观察重点

比较 CUDA 和 CUB 时，可以重点看：

- `N` 较大时，CUB 分段归约是否更能利用 GPU 并行。
- `N` 较小时，中间数组写入和 CUB 调用开销是否抵消收益。
- CUB 的 kernel 时间是否更稳定。
- 分阶段设计带来的额外内存流量是否明显。

这个版本特别适合用来解释：工业级原语很强，但算子融合和减少中间写入同样重要。
