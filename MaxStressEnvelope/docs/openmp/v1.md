# MaxStressEnvelope / OpenMP / v1

## 当前实现

OpenMP v1 保留 baseline 的单元内工况遍历，只把最外层单元循环并行化：

```cpp
#pragma omp parallel for
for (int i = 0; i < M; ++i) {
    float max_vm = 0.0f;
    for (int j = 0; j < N; ++j) {
        int idx = i * N + j;
        float ds = sx[idx] - sy[idx];
        float ss = sy[idx] - sz[idx];
        float ts = sz[idx] - sx[idx];
        float vm = std::sqrt(0.5f * (ds*ds + ss*ss + ts*ts +
                                     6.0f * (txy[idx]*txy[idx] + tyz[idx]*tyz[idx] + tzx[idx]*tzx[idx])));
        max_vm = std::max(max_vm, vm);
    }
    envelope[i] = max_vm;
}
```

源码位置：`MaxStressEnvelope/kernels/openmp.cpp`

## 优化程度

这是 v1 的 CPU 多线程实现。它没有改变数据布局，也没有拆分工况归约，只把不同单元分配给多个线程。

| 方向 | 当前状态 |
|---|---|
| 线程级并行 | `#pragma omp parallel for` 按单元并行 |
| 单元内归约 | 每个线程内部串行 max |
| 数据重排 | 无 |
| 调度策略 | OpenMP 默认调度 |
| 适合目标 | 观察多核 CPU 对多工况包络的加速 |

## 优化思路

每个输出 `envelope[i]` 只依赖第 `i` 个单元的所有工况，不依赖其他单元。因此单元之间天然独立。

OpenMP v1 把单元循环分给多个 CPU 线程：

- 每个线程处理若干个单元。
- 每个线程在自己的局部变量 `max_vm` 中完成工况归约。
- 不同线程写不同的 `envelope[i]`，不需要锁或原子操作。

## 性能瓶颈

这个版本的加速效果取决于 `M` 和 `N`：

- `M` 较小时，可并行的单元数量不足，线程利用率有限。
- `N` 较大时，每个单元内部串行工作变多，单线程内层循环更重。
- 输入数组连续访问，但总数据量是 `6 * M * N` 个 float，容易受内存带宽影响。
- 每个工况都有 `sqrt`，数学函数吞吐也会影响性能。

## 观察重点

OpenMP 版本适合观察“按单元并行”是否能吃满 CPU 多核。和 Von Mises 不同，这里每个输出背后有一个小归约，`N` 越大，单个线程持有的工作越重。
